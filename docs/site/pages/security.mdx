# Security

DURANDAL is designed with a defense-in-depth security architecture. Because the platform executes AI-generated code and interacts with external services on your behalf, every layer of the system enforces isolation, access control, and auditability.

## 5-Layer Defense-in-Depth

DURANDAL applies security at five distinct layers:

| Layer | Protection | Implementation |
| ----- | ---------- | -------------- |
| 1. Container Isolation | Sandboxed execution | Docker containers with dropped capabilities and `no-new-privileges` |
| 2. Network Control | Scoped connectivity | Per-step network allowlists restrict which domains each tool call can reach |
| 3. Credential Vault | Secret protection | AES-256-GCM encryption with scrypt key derivation |
| 4. Access Control | User permissions | Three-tier RBAC (Owner > Admin > Member) on all API endpoints |
| 5. Audit Trail | Accountability | Every action logged with actor, timestamp, resource, and details |

These layers work together so that a compromise at any single layer is contained by the others.

## Container Isolation

Every tool execution in DURANDAL runs inside an isolated Docker container managed by NanoClaw. This is the primary defense against AI-generated code executing unintended actions.

### Security Controls

Each container is launched with:

- **`no-new-privileges: true`** -- Prevents privilege escalation via setuid/setgid binaries
- **`cap_drop: ALL`** -- Drops all Linux capabilities. The container runs with the absolute minimum kernel access.
- **Per-step network allowlists** -- Each template step declares exactly which domains it needs. The container network is configured to only allow connections to those domains. Steps with an empty allowlist have no network access at all.
- **Timeout enforcement** -- Each step has a configurable timeout (typically 60-180 seconds). If execution exceeds the timeout, the container is killed.
- **Fresh containers** -- Each step runs in a new container. No state leaks between steps except through the explicit step results passed by the orchestrator.

### Network Allowlists

The network allowlist system prevents the AI agent from exfiltrating data to unexpected destinations. Allowlists are defined at two levels:

1. **Step-level** -- Each step in a template declares its own `networkAllowlist`. For example, a "fetch tickets" step might only allow `{{helpdesk_endpoint}}`, while a "classify tickets" step might have an empty allowlist (no network).

2. **Template-level** -- A top-level `networkAllowlist` on the template provides baseline domains available to all steps.

Allowlists support glob patterns: `*.gmail.com`, `smtp.*`, `api.example.com`.

### What This Prevents

- **Data exfiltration** -- The AI cannot send your data to unauthorized servers because the network allowlist blocks all undeclared destinations
- **Supply chain attacks** -- Containers cannot download or execute arbitrary packages from the internet
- **Lateral movement** -- Containers are isolated from the host network and from each other
- **Persistent compromise** -- Fresh containers mean malicious state cannot persist between executions

## Credential Vault

API keys, SMTP passwords, and other secrets are stored in an encrypted vault using the `@durandal/vault` package.

### Encryption Details

| Property | Value |
| -------- | ----- |
| Algorithm | AES-256-GCM (authenticated encryption) |
| Key derivation | scrypt with 32-byte salt |
| IV | 16 bytes, randomly generated per encryption |
| Auth tag | 128-bit GCM authentication tag |
| Storage format | `salt:iv:tag:ciphertext` (hex-encoded, colon-separated) |
| Master key source | `VAULT_MASTER_KEY` env var, or `NEXTAUTH_SECRET` as fallback |

### How It Works

1. When an admin stores a credential via `POST /api/credentials`, the plain-text value is immediately encrypted using `encrypt(value, masterKey)`.
2. The encrypted string (containing salt, IV, auth tag, and ciphertext) is stored in the SQLite `credentials` table. The IV is also stored in a separate column for indexing purposes.
3. The plain-text value is never written to disk, logged, or returned by any API endpoint.
4. When a template step needs a credential, the system decrypts it in memory, injects it into the container environment, and the credential exists in plain text only for the duration of that container's execution.

### What This Prevents

- **Database theft** -- If the SQLite database file is stolen, credentials cannot be decrypted without the master key
- **Log leakage** -- Plain-text credentials never appear in logs or API responses
- **Memory persistence** -- Credentials are decrypted only during active container execution

## Audit Logging

Every significant action in DURANDAL is recorded in the `audit_log` database table, creating a tamper-evident record of system activity.

### Logged Events

| Action | Description |
| ------ | ----------- |
| `task.create` | A new task was created |
| `task.cancel` | A task was cancelled |
| `template.run` | A template execution was started |
| `template.approval_required` | A step paused for human approval |
| `template.step_failed` | A step execution failed |
| `template.completed` | All template steps completed successfully |
| `approval.approve` | A pending approval was approved |
| `approval.reject` | A pending approval was rejected |
| `credential.create` | A new credential was added to the vault |
| `credential.delete` | A credential was removed |
| `schedule.create` | A new schedule was created |
| `schedule.update` | A schedule was modified |
| `schedule.delete` | A schedule was removed |
| `memory.update` | The agent's memory files were modified |
| `user.create` | A new user account was created |

### Audit Entry Structure

Each audit entry contains:

```json
{
  "id": "unique-uuid",
  "timestamp": "2026-04-01T09:05:00.000Z",
  "actor": "admin@company.com",
  "action": "credential.create",
  "resource": "credential:cred-uuid",
  "details": "zendesk/helpdesk_api_key"
}
```

- **actor** -- The email or user ID of who performed the action
- **action** -- A dot-separated action type
- **resource** -- The resource type and ID affected
- **details** -- Human-readable context

The audit log is accessible via `GET /api/audit` (returns the 100 most recent entries) and directly from the SQLite database for compliance exports.

## RBAC Model

DURANDAL uses a three-tier Role-Based Access Control system:

### Roles

| Role | Level | Description |
| ---- | ----- | ----------- |
| **Owner** | Highest | Full system control. Created during initial setup. Can manage licenses, telemetry, all users, and all system settings. |
| **Admin** | Middle | Can manage credentials, approve/reject tasks, manage schedules, update agent memory, and manage member accounts. |
| **Member** | Lowest | Can view tasks and templates, run templates, view credentials (names only), and view the audit log. Cannot modify system configuration. |

### Permission Matrix

| Action | Owner | Admin | Member |
| ------ | ----- | ----- | ------ |
| Activate license | Yes | No | No |
| Configure telemetry | Yes | No | No |
| Manage users | Yes | Yes | No |
| Add/delete credentials | Yes | Yes | No |
| Update agent memory | Yes | Yes | No |
| Approve/reject tasks | Yes | Yes | No |
| Manage schedules | Yes | Yes | Yes |
| Run templates | Yes | Yes | Yes |
| View tasks & logs | Yes | Yes | Yes |
| View credential names | Yes | Yes | Yes |

### Implementation

RBAC is enforced at the API layer using the `requireRole()` middleware in `apps/dashboard/src/lib/rbac.ts`. Each API route declares the minimum role required. If the authenticated user's role is insufficient, the API returns a 403 Forbidden response.

The first user created via the setup wizard automatically receives the Owner role. Subsequent users can be assigned Owner, Admin, or Member roles by existing Owners or Admins.

## Air-Gap Mode

DURANDAL is designed to run fully offline (air-gapped) for environments where internet access is not available or not permitted.

### What Works Offline

- **Local model inference** -- Ollama runs locally and models can be loaded from local files or USB storage
- **All core features** -- Task execution, template processing, approvals, scheduling, credential management, and audit logging all work without internet
- **The dashboard UI** -- Served locally from the container, no CDN dependencies

### What Requires Internet

- **Cloud LLM providers** -- If using external AI providers (Pro/Enterprise license feature), those API calls need internet
- **Template steps with network allowlists** -- Steps that interact with external APIs need connectivity to those specific endpoints
- **Telemetry** -- Optional and disabled by default; requires internet to send data

### Air-Gap Deployment

For air-gapped environments:

1. Pre-build Docker images on a connected machine
2. Export images with `docker save`
3. Transfer images via USB or local network
4. Load images with `docker load`
5. Pre-download Ollama models and copy them to the `ollama-models` volume

See the [Deployment guide](/deployment) for detailed air-gap instructions.

## Prompt Injection Defense

Because DURANDAL executes AI-generated plans, prompt injection is a key threat vector. The platform defends against this at multiple levels:

### Structural Defenses

1. **Template-defined boundaries** -- Templates constrain what the AI can do. Each step has a specific prompt, a network allowlist, and a timeout. The AI cannot escape the template structure.

2. **Parameter substitution only** -- User input is injected into prompts via `{{paramName}}` substitution, not by appending raw user input to system prompts. This reduces the attack surface for indirect injection.

3. **Approval gates** -- Steps that perform external actions (sending emails, posting data, updating systems) require human approval. Even if an AI is manipulated by injected content, the human reviewer can catch unexpected actions before they execute.

4. **Network allowlists** -- Even if an injection causes the AI to attempt data exfiltration, the container network restrictions prevent connections to unauthorized destinations.

5. **Container isolation** -- If an injection causes the AI to generate malicious code, that code runs in a sandboxed container with no capabilities, no privilege escalation, and a strict timeout.

### Defense Layers Summary

An attacker attempting prompt injection would need to:
1. Craft a payload that survives parameter substitution
2. Convince the AI to generate malicious actions
3. Get past the human approval gate (for sensitive steps)
4. Bypass the network allowlist (for data exfiltration)
5. Escape the container sandbox (for host compromise)

Each layer is independent, making a successful end-to-end attack extremely difficult.

## Security Best Practices

When operating DURANDAL:

- **Set a strong `NEXTAUTH_SECRET`** -- Use `openssl rand -base64 32` to generate a cryptographically random secret. This is used for session signing and as a fallback for credential encryption.
- **Set `VAULT_MASTER_KEY` separately** -- Do not rely on the NEXTAUTH_SECRET fallback for production credential encryption. Generate a dedicated vault key.
- **Minimise network allowlists** -- Only allow the domains each step actually needs. Use empty allowlists for computation-only steps.
- **Enable approval gates for external actions** -- Any step that sends data to external systems should have `requiresApproval: true`.
- **Review audit logs regularly** -- Check for unexpected actions, failed authentication attempts, or unusual patterns.
- **Keep Docker updated** -- Container security depends on the Docker runtime. Run the latest stable version.
- **Restrict dashboard access** -- Use a firewall or VPN to limit who can reach the dashboard. By default, Caddy handles HTTPS on ports 443/80.
